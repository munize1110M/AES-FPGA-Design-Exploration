
## Intro/Background
- The AES-128 is a symmetric block cipher established by the National Institute of Standards and Technologies that encrypts data in 128-bit blocks using a 128-bit key over 10 processing rounds. 
- ### AES-128 Key Features
	- Symmetric Key: The same 128-bit key is used for both encryption and decryption
	- Block size: Operates on  128-bit (16-byte) blocks of data, organized into a 4x4 state matrix
		- can also be 192 bit and 256 bit
	- Rounds: The cipher performs 10 rounds of transformation for key expansion and data encryption
	- ###### Modes:
		- While here I am simply implementing the Cipher in hardware an analyzing the implementations, its important to be familiar with the larger picture in regard to its applications
		- ###### CBC (Cipher Block Chaining)
			- Each block of plaintext is XORed with the previous ciphertext block before being encrypted, ensuring each block depends on all previous ones
		- ###### CTR (Counter Mode)
			- Turns a block cipher into a stream cipher by encryption a successive counter
			- Nonce + Counter values gets sent to be encrypted then bit wise xor'ed with the plaintext
		- ###### GCM (Galois/Counter Mode)
			- Modern, high-performance mode providing both confidentially and authentication
		- What each of these modes have in common is the that they must perform encryption at some point regardless of what gets encrypted and what happens outside of the cipher. 
	
- ### How does the Cipher Work
	- First (depending on implementation) a key expansion must be performed during which 10 individual keys are derived from the initial encryption key. Each key corresponds to a single round in the cipher. While my implementation has the key expansion pre loaded in ROM, some implementations may choose to compute relevant round keys during the cipher.
	- S-box: The Substitution-Box is a non linear component that substitutes input bytes with new values using a fixed 16x16 lookup table. Once again pre generated and stored as a LUT in my implementation.
		- The sbox is generated by calculating the multiplication inverse in the finite field $GF(2^8)$ using the irreducible polynomial $x^8+x^4+x^3+x+1$ followed by an affine transformation to increase algebraic complexity.
		- The goal of using the Sbox is to minimize the correlation between input and output to defeat linear cryptanalysis.

		- ##### AddRound Key (Initial Round)
			- Plaintext is combined with the original encryption key using a bitwise XOR operation
		- ##### Main Rounds (1-9)
			- ##### SubBytes
				- Each byte is replaced with another from a fixed LUT (S-box) to provide confusion
			- ##### Shift Rows
				- The rows of the $4x4$ state matrix are shifted cyclically to the left by different offsets (0, 1, 2, 3)  bytes (dependent on row number)
			- ##### MixColumns
				- A mathematical formula mixes the data in each column to ensure every byte affects multiple @ the output.
				- In my implementation, this was done using various bit operations which happened to be particularly efficient

## Python Golden Model
- #### Why did I make this?
	- I began my design process by first making a python golden model. The goal was the emulate the hardware behavior in order to provide a working reference to test my hardware implementation. Additionally, making the python golden model (while simultaneously emulating the hardware behavior) ensured that I was able to fully understand the inner workings of the AES Cipher.
- #### How does it work
	- The simulation is comprised of a file `aes_128_FIPS.py`. The simulation contains key functions such as `sub_bytes()`, `shift_rows()`, `mix_columns()`, and `add_round_key()`
	- These functions were made in order to emulated the behavior and structure of a SystemVerilog based Cipher implementation.
	- In addition to these functions, the `S-box` and `key_expansion()` perform the necessary substitutions and key operations to begin the cipher.
	- The original 128-bit plain text and 128-bit encryption key are hardcoded in the `__main__` function. The 128-bit values are as follows
		- Original Plaintext: `0x_3243f6a8885a308d313198a2e0370734`
		- Encryption Key: `0x_2b7e151628aed2a6abf7158809cf4f3c`
		- These values come from Appendix B of the FIPS 197 AES Spec. This example is provided to in order to aid in verifying AES implementations.
	- Using these values, the simulation runs and also prints the State Matrix at EVERY step of the cipher. This is important so as to verify the hardware cipher implementation.

## Comparison & Discussion
- Vivado provides some neat insights on each of the implementations. What is immediately evident is that the resource usage for the Fully Pipelined implementation significantly increased when compared to the initial "blocking" version. To be more specific, LUT resource usage jumped from 2.53% utilization (1607 LUTs) to 10.78% utilization (6832 LUTs). In addition to the LUT usage, Flip Flop (FF) usage also increased from 0.73% utilization to 1.43% utilization.
- In addition to the resource usage, the estimated power usage more than doubles when comparing the two implementations. The increase of power largely comes from the increase in `Logic` Power Consumption.
- The increase resource and power utilization was expected when jumping from the initial version to the fully pipelined version. Focusing on the increase in FF usage, this is directly due to the introduction of stages and pipeline registers. The original implementation featured register banks that were reused during the entirety of the 10 rounds and the data flow was managed by a state machine. 
- However, the fully pipelined version didn't reused stages. Instead, it "unrolled" the cipher and introduced 10 stages (1 stage for each round). 10 stages means more pipeline registers will be needed (4 more register stages compared to the initial implementation).
- The increase in LUT usage is a result of each pipeline stage having its own copy of the SBOX LUT. This is necessary in order to maintain the 10 cycle target latency for this implementation.


- {discuss that the original implementation splits up each step within a round into stages and then resuses these registers for each of the 10 cipher rounds, whereas, the new implementation, completes an entire round (add key, sub bytes, shift rows, mix columns) in one cycle) thus leading to 10 cycles for one cipher}
- Initial Vivado timing shows that both designs pass timing for a Nexys A7 -100T Target. What is evident is that we have more "clock margin" in our Original Implementation when compared to the Pipelined Version. This is expected because in the original version, the individual steps of each round were essentially pipelined, whereas the pipelined version has each round happening in one clock cycle. 
	- This tradeoff is definitely something to explore for specific applications. (power, resource usage, slack, latency + throughput, etc)

## Conclusion and Future Work
- To sum up this design exploration, what is evident is that the fully pipelined version offers higher performance when compared to the original version at the cost increased resource usage and increased power usage. While this result was expected, there were notable things discovered as a result of this exploration. What I found especially interesting is that while both designs passed timing with great margin to spare, the original version was only marginally better. For many applications, this means that choosing the fully pipelined version is a no brainer.
- There are many ways this project can be expanded and optimized. First, this project features a SPI wrapper that has been tested an verified using a Arduino MKR WiFi 1010 as a SPI master. Further system analysis can be done with multiple SPI masters in order to aim to keep the Pipeline busy at all times. 
- Additionally, while I acknowledge that using SPI to send data to a master module introduces a massive bottleneck, it made bring up very easy and allowed me to verify this design in a real world application. For an actual high speed target, I would be looking at using a different dev board and not the Nexys A7 100T.
- I would also like to look into optimizations in the future. While compiling all the resources necessary to complete this report and repository, I noticed a few areas in the code where I may be inferring a latch. This is just one example of where the code could be further optimized. 
- Finally, I'd like to see the maximum clock speed each of these designs can achieve. Simply using the Worst Negative Slack for each of the designs, the Pipelined Version will likely fail before the original version.
