# AES-128 Hardware Design Exploration

This repository contains an Advanced VLSI Design final project exploring two hardware implementations of an AES-128 encryption core targeting the Nexys A7-100T FPGA development board. The project compares a register-reuse, blocking implementation against a fully pipelined implementation in terms of latency, throughput, resource utilization, power, and timing.

## Project Overview

AES-128 is a symmetric block cipher standardized by NIST. It encrypts 128-bit blocks using a 128-bit key over 10 rounds of transformation. This project implements the AES cipher in hardware, validates the implementation against a Python golden model, and compares two SystemVerilog architectures.

Key AES-128 properties:

- **Symmetric key:** the same 128-bit key is used for encryption and decryption.
- **Block size:** AES operates on 128-bit blocks organized as a 4x4 state matrix.
- **Rounds:** AES-128 performs 10 rounds of transformation.
- **Common modes:** AES is commonly used in CBC, CTR, and GCM modes. This project focuses on the cipher core itself.

## AES Cipher Background

The AES encryption flow consists of an initial AddRoundKey step, 9 main rounds, and a final round. The main rounds perform SubBytes, ShiftRows, MixColumns, and AddRoundKey. The final round omits MixColumns.

![AES cipher flow](Images/Pasted%20image%2020260429184639.png)

### Core Operations

- **AddRoundKey:** XORs the state matrix with the current round key.
- **SubBytes:** replaces each byte using a fixed S-box lookup table.
- **ShiftRows:** cyclically shifts each row of the state matrix by a row-dependent offset.
- **MixColumns:** mixes each column so that each byte influences multiple output bytes.
- **Key expansion:** derives the per-round keys from the original 128-bit encryption key.

The S-box is generated by calculating the multiplicative inverse in the finite field \(GF(2^8)\) using the irreducible polynomial \(x^8 + x^4 + x^3 + x + 1\), followed by an affine transformation.

## Python Golden Model

Before implementing the hardware design, a Python golden model was created to emulate the AES hardware behavior and provide a reference for verification.

The model is contained in `aes_128_FIPS.py` and includes functions such as:

- `sub_bytes()`
- `shift_rows()`
- `mix_columns()`
- `add_round_key()`
- `key_expansion()`

The test vector uses the AES example from Appendix B of the FIPS 197 specification:

- Plaintext: `0x3243f6a8885a308d313198a2e0370734`
- Encryption key: `0x2b7e151628aed2a6abf7158809cf4f3c`

The simulation prints the state matrix at each step of the cipher to support hardware verification.

## FPGA Target

This design targets the **Nexys A7-100T FPGA development board**.

The design was successfully synthesized and implemented at **100 MHz**. The implementation was also validated in a real application: real-time encryption of sensor data for an IoT edge use case.

For the real FPGA demonstration, the AES core was wrapped with an SPI core so the Nexys A7 PMOD ports could interface with a co-processor over SPI. Because SPI is part of the FPGA target design, it is included in the resource and power calculations.

## Implementation 1: Blocking AES-128 Core

The first implementation reuses register banks across the 10 AES rounds. The data flow is controlled by a system state machine, and the implementation produces one ciphertext every 30 cycles.

- Latency: **30 cycles**
- Throughput: **1 ciphertext every 30 cycles**
- Target clock: **100 MHz**

### Architecture

The original architecture pipelines individual operations within each round, such as SubBytes and MixColumns, while reusing the same register banks across all rounds.

## Implementation 2: Fully Pipelined AES-128 Core

The second implementation fully unrolls the AES cipher into pipeline stages. Each stage performs one AES round, allowing the design to accept a new plaintext block every cycle after the initial pipeline fill.

- Latency: **10 cycles**
- Throughput: **1 ciphertext every cycle after startup**
- Target clock: **100 MHz**

## Comparison and Discussion

The fully pipelined implementation significantly increases performance, but it also increases area and power consumption.

LUT utilization increased from **2.53%** in the blocking implementation to **10.78%** in the fully pipelined implementation. In absolute terms, LUT usage increased from **1,607 LUTs** to **6,832 LUTs**.

Flip-flop utilization also increased from **0.73%** to **1.43%**. This increase is expected because the fully pipelined implementation introduces additional stage registers across the unrolled AES rounds.

Power consumption more than doubled, primarily due to increased logic power. This is also expected because each pipeline stage has its own logic resources, including S-box lookup logic, rather than reusing the same round hardware across multiple cycles.

The blocking implementation reuses a smaller amount of hardware over 30 cycles. The fully pipelined version unrolls the cipher so each AES round completes in one cycle, reducing latency to 10 cycles and allowing one new block to be processed every cycle once the pipeline is full.

Both designs pass timing on the Nexys A7-100T target at 100 MHz. The blocking implementation has slightly more timing margin because each cycle contains less round-level combinational logic.

### Throughput

Blocking implementation:

```text
Throughput = (128 bits / 30 cycles) * 100 MHz
Throughput = 426,666,666.67 bits/s
Throughput = 426.67 Mbps = 53.33 MB/s
```

Fully pipelined implementation:

```text
Throughput = 128 bits * 100 MHz
Throughput = 12,800,000,000 bits/s
Throughput = 12.8 Gbps = 1.6 GB/s
```

## Conclusion and Future Work

The fully pipelined AES-128 core provides substantially higher throughput than the blocking implementation, at the cost of increased LUT usage, flip-flop usage, and power consumption. For many applications, the throughput improvement makes the fully pipelined implementation the better choice, especially since both designs pass timing at 100 MHz on the Nexys A7-100T.

Future work includes:

- Further system-level analysis with multiple SPI masters to keep the pipeline busy.
- Replacing SPI with a higher-speed interface for applications where communication bandwidth is the bottleneck.
- Optimizing areas of the SystemVerilog implementation that may infer unintended latches.
- Exploring the maximum achievable clock frequency for each architecture.
- Comparing additional tradeoffs between latency, throughput, power, timing margin, and resource utilization.
